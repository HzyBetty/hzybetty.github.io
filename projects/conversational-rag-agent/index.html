The problem

GitHub's documentation is vast and scattered. Users waste time navigating docs for routine tasks — setting up 2FA, managing billing, creating repositories — while support teams handle repetitive queries that could be automated.

This project builds a single conversational interface that handles the full range of GitHub support needs: answered questions backed by cited documentation, support ticket lifecycle management, and CRM operations (user registration, billing, plan upgrades) — all with safety guardrails to prevent misuse.

RAG knowledge Q&A

User questions are routed to a vector retriever backed by scraped GitHub docs. The LLM generates answers grounded in retrieved chunks, with source URLs cited inline. A collapsible "Why this answer?" panel exposes the top-5 retrieved chunks with cosine similarity scores, giving full transparency into the retrieval process.

Query: "How do I set up two-factor authentication on GitHub?" — 5 chunks retrieved, top match at 93% similarity.

Architecture

Every user message passes through an intent router (regex-first, LLM fallback) that dispatches to one of nine specialised handlers. This keeps each capability isolated and testable while sharing a single conversation memory and SQLite persistence layer.

Layer 01
Ingestion
GitHub docs scraped, chunked, and embedded into ChromaDB using all-MiniLM-L6-v2
Layer 02
Routing
Regex patterns cover common intents; LLM fallback handles ambiguous phrasing
Layer 03
Generation
Retrieved chunks passed to Llama-3.1 via Groq API; answer grounded with cited sources
Layer 04
Persistence
Tickets, user accounts, and session history stored in SQLite for cross-session continuity

Support ticket lifecycle

Ticket creation uses a state machine with four sequential turns: category selection, issue description, priority triage, and a review-before-submit step. Created tickets are persisted in SQLite and visible in the sidebar. Checking a ticket by ID optionally triggers a RAG lookup to provide guidance based on the issue description.

Multi-turn ticket creation and lookup
Category → description → priority → confirm → ticket ID assigned. Follow-up lookup retrieves RAG-guided resolution steps.

CRM operations

The agent doubles as a lightweight CRM: it handles user registration with plan selection, billing lookups, plan upgrades with a before/after comparison, and full account listing. All state is persisted to SQLite and reflected immediately across turns.

Registration, plan upgrade, and account listing
User 'lucy' not found → guided registration on Free plan → upgraded to Team → account list confirms all 6 registered users.

Safety & guardrails

Two guardrail types are enforced at the intent routing layer. Prompt injection attempts are detected and blocked before reaching the LLM. Out-of-scope requests (non-GitHub topics) return a redirect message. When the knowledge base cannot reliably answer a question, the agent returns an "insufficient evidence" response rather than hallucinating — and suggests opening a support ticket if appropriate.

Prompt injection block, out-of-scope redirect, and knowledge gap handling
Three guardrail scenarios in one session: injection attempt blocked, weather question redirected, future roadmap question returns honest uncertainty.

Analytics dashboard

A separate Streamlit page tracks live session telemetry. Intent distribution shows how users are engaging with the agent; a confidence-over-time chart tracks answer quality across a session; a knowledge gap log surfaces questions the retriever couldn't adequately answer — a direct signal for doc coverage improvements.

Three evaluation suites were run against the agent. The baseline suite tests expected intent routing and answer quality on curated prompts. The stress suite covers edge cases: typos, ambiguous phrasing, multi-intent queries, and adversarial inputs.

Future directions

Several improvements could extend performance and scope:

  • Hybrid retrieval — combine dense (vector) and sparse (BM25) search for better coverage on rare queries
  • Re-ranking — add a cross-encoder re-ranker between retrieval and generation to improve top-chunk precision
  • Multi-doc ingestion — extend beyond GitHub docs to organisation wikis or internal knowledge bases
  • Human-in-the-loop escalation — route low-confidence answers to a human agent rather than returning uncertainty
  • Automated doc gap closure — use the knowledge gap log to trigger targeted scraping of missing documentation
